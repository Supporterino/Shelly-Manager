# AGENTS.md — AI Agent Guide for Shelly Manager

This file tells any AI coding agent (OpenCode, Cursor, Copilot, etc.) everything it needs to work effectively in this codebase. Read this before making any changes.

---

## Project Summary

**Shelly Manager** is a local-network Shelly smart-device manager.
It runs on iOS, Android, macOS, Windows, and Linux.
It communicates with Shelly Gen 2/3/4 devices via their JSON-RPC 2.0 HTTP API.

Full build plan: see [PLAN.md](./PLAN.md).

## Absolute Rules

1. **Never use `window.fetch` or `axios` for Shelly API calls.** Always use `fetch` from `@tauri-apps/plugin-http`. It works on mobile; the browser fetch does not reach LAN addresses on iOS/Android.

2. **Never hard-code device IPs.** All device data comes from the Zustand `deviceStore` which is hydrated from `tauri-plugin-store`.

3. **Never edit `src/routeTree.gen.ts`** — it is auto-generated by the TanStack Router Vite plugin on every save. Editing it will be overwritten.

4. **Never use `BrowserRouter` or `createBrowserHistory`.** Tauri has no server to handle HTML5 history rewrites. Always use `createHashHistory()`.

5. **All Rust logic lives in `src-tauri/src/lib.rs`.** Never modify `src-tauri/src/main.rs` — it is a 3-line entry point that calls `lib::run()`.

6. **Keep React components under ~120 lines.** If a component grows beyond this, split it into smaller sub-components.

7. **Write tests alongside every new service or utility.** Tests live as `*.test.ts` next to the source file or in `src/__tests__/`.

8. **Never use raw string literals for user-visible text.** All UI copy must go through `useTranslation` and live in `src/locales/en/*.json`. Add the key to the English file first, then translate.

9. **Never use `i18next-http-backend`.** Tauri's WebView cannot serve files over HTTP. Use `i18next-resources-to-backend` with Vite dynamic `import()` exclusively.

## Shelly API Reference

### Supported generations

Only **Gen 2, Gen 3, Gen 4** are supported. These use the **JSON-RPC 2.0** protocol.

Gen 1 (REST) is NOT supported.

### How to call a Gen2+ device

**HTTP (one-shot):**
```
POST http://<device-ip>/rpc
Content-Type: application/json

{ "id": 1, "src": "shelly-manager", "method": "Switch.Set", "params": { "id": 0, "on": true } }
```

**Shorthand GET:**
```
GET http://<device-ip>/rpc/Switch.Set?id=0&on=true
```

**WebSocket (persistent, for real-time):**
```
ws://<device-ip>/rpc
```

### Key RPC methods

| Method | Description |
|---|---|
| `Shelly.GetDeviceInfo` | Device identity (MAC, model, gen, auth_en). Never requires auth. |
| `Shelly.GetStatus` | Full status of all components at once |
| `Shelly.GetConfig` | Full config of all components |
| `Shelly.SetAuth` | Enable/disable auth + set password |
| `Shelly.Reboot` | Reboot the device |
| `Shelly.FactoryReset` | Full factory reset |
| `Shelly.CheckForUpdate` | Check firmware update availability |
| `Shelly.Update` | Trigger OTA firmware update |
| `Switch.Set` | `{ id, on }` — turn a relay on/off |
| `Switch.GetStatus` | `{ id }` → `{ output, apower, voltage, current, aenergy }` |
| `Light.Set` | `{ id, on, brightness }` — dimmer control |
| `Cover.Open/Close/Stop` | `{ id }` — blind/cover control |
| `Schedule.List` | List device schedules |
| `Schedule.Create` | Create a schedule |
| `Schedule.Delete` | `{ id }` — delete a schedule |

### Authentication (Gen2+)

Auth is SHA-256 Digest (RFC 7616). Implementation is in `src/utils/auth.ts`.

Flow:
1. Send request without auth
2. If HTTP 401 received, extract `nonce`, `realm` from `WWW-Authenticate`
3. Compute `ha1 = sha256("admin:<realm>:<password>")`
4. Compute `ha2 = sha256("dummy_method:dummy_uri")`
5. Compute `response = sha256("<ha1>:<nonce>:<nc>:<cnonce>:auth:<ha2>")`
6. Resend request with `auth` object in the JSON-RPC frame

Over WebSocket: authenticate once, reuse `auth` object for the session lifetime.

`Shelly.GetDeviceInfo` and `GET /shelly` are always exempt from auth.

### Notifications (WebSocket only)

After sending one request with `src: 'shelly-manager'`, the device starts pushing:
- `NotifyStatus` — partial status delta (deep-merge into current state)
- `NotifyFullStatus` — full status (replace current state)
- `NotifyEvent` — discrete events (button press, config change, etc.)

### Discovery

| Method | How |
|---|---|
| mDNS | Rust command: browse `_shelly._tcp.local.` (Gen2+ only) and `_http._tcp.local.` with `gen=2/3/4` TXT filter |
| Subnet scan | Rust command: TCP connect to port 80 on each IP in CIDR, then verify with `GET /shelly` |
| Manual | User enters IP; frontend verifies with `GET /shelly` |

`GET http://<ip>/shelly` returns `{ type, mac, auth, fw, gen }` — use this to confirm a device and get its MAC/gen before adding.

## Zustand Store Conventions

### Access pattern

```ts
// Always use selectors — never destructure the whole store
const devices = useDeviceStore(s => s.devices)
const addDevice = useDeviceStore(s => s.addDevice)
```

### Persistence pattern

`deviceStore` and `appStore` persist via `tauri-plugin-store`. Call `hydrate()` once in `__root.tsx` on mount. The store auto-calls `persist()` via a `subscribeWithSelector` subscription on every write.

**`tauri-plugin-store` v2 JS API:** Use `Store.load('file.json')` (static async factory), **not** `new Store(...)`. The store auto-saves on `set()` by default.

```ts
import { Store } from '@tauri-apps/plugin-store'
const store = await Store.load('shelly-manager.json')
await store.set('devices', devicesRecord)
const devices = await store.get<Record<string, StoredDevice>>('devices')
```

### Store files

| Store | Contents |
|---|---|
| `deviceStore.ts` | `devices: Record<id, StoredDevice>` + CRUD |
| `wsStatusStore.ts` | `statuses: Record<id, ShellyStatus>` + `connected: Record<id, bool>` |
| `appStore.ts` | `preferences: AppPreferences` (theme, polling interval, etc.) |

## TanStack Query Conventions

### Query keys

Always follow this hierarchy:
```ts
['device', deviceId, 'status']    // Shelly.GetStatus for one device
['device', deviceId, 'config']    // Shelly.GetConfig
['device', deviceId, 'info']      // Shelly.GetDeviceInfo
['devices', 'all', 'status']      // bulk poll for dashboard
```

### Mutations

After any successful mutation that changes device state, invalidate the relevant query:
```ts
onSuccess: () => queryClient.invalidateQueries({ queryKey: ['device', deviceId, 'status'] })
```

Show an error notification on failure:
```ts
onError: (err) => notifications.show({ color: 'red', title: 'Error', message: err.message })
```

## TanStack Router Conventions

### Route files location

All routes live in `src/routes/`. The plugin watches this directory and auto-generates `routeTree.gen.ts`.

### File naming

| File | Route |
|---|---|
| `__root.tsx` | Root layout |
| `index.tsx` | `/` |
| `discover.tsx` | `/discover` |
| `settings.tsx` | `/settings` |
| `devices/$deviceId.tsx` | `/devices/:deviceId` |
| `devices/$deviceId.settings.tsx` | `/devices/:deviceId/settings` |

### Route definition pattern

```ts
export const Route = createFileRoute('/devices/$deviceId')({
  loader: async ({ params, context }) => {
    // Pre-fetch device info (runs before component renders)
    return shellyClient.getDeviceInfo()
  },
  component: DeviceDetailPage,
  errorComponent: DeviceErrorPage,
})

function DeviceDetailPage() {
  const { deviceId } = Route.useParams()
  const info = Route.useLoaderData()
  // ...
}
```

### Navigation

```tsx
import { Link, useNavigate } from '@tanstack/react-router'

// Declarative
<Link to="/devices/$deviceId" params={{ deviceId: device.id }}>Open</Link>

// Programmatic
const navigate = useNavigate()
await navigate({ to: '/devices/$deviceId', params: { deviceId } })
```

## Rust Command Conventions

### Adding a new command

1. Define the function in `src-tauri/src/lib.rs` with `#[tauri::command]`
2. Add it to `tauri::generate_handler![...]` in `pub fn run()`
3. Call it from the frontend with `invoke('command_name', { param: value })`

### Return types

Always return `Result<T, String>` where the error string is a human-readable message:
```rust
#[tauri::command]
async fn my_command(arg: String) -> Result<SomeType, String> {
  some_operation(arg).map_err(|e| e.to_string())
}
```

### Serializable types

Use `serde::Serialize` and `serde::Deserialize` on all types passed between Rust and JS:
```rust
#[derive(Debug, Serialize, Deserialize)]
pub struct DiscoveredHost {
  pub ip: String,
  pub port: u16,
  pub hostname: Option<String>,
  pub source: String,
}
```

## i18n Usage Guide

### Library: react-i18next (i18next v26)

Translation files live in `src/locales/<lang>/<namespace>.json`. The four namespaces map directly to the app's route structure:

| Namespace | Loaded by | Key content |
|---|---|---|
| `common` | All routes | Actions, status labels, error messages |
| `devices` | Dashboard + device detail | Type names, control labels, info, firmware |
| `discovery` | `/discover` | Wizard copy, form labels, plural counts |
| `settings` | `/settings` | Theme options, language names, about |

### Hook usage

```tsx
// Always use the namespace that matches the current route
const { t } = useTranslation('devices')

// Simple key
t('controls.on')                        // → "On"

// Interpolation
t('firmware.updateAvailable', { version: '1.4.2' })   // → "Update available: 1.4.2"

// Plurals — i18next selects _one / _other automatically
t('found', { count: 3 })               // → "Found 3 devices"

// Cross-namespace reference
t('common:actions.save')
```

### Number / date formatting

Use the native `Intl` API with `i18n.language` — do NOT hardcode locale strings:

```ts
const { i18n } = useTranslation()

// Watts with locale-correct decimal separator
new Intl.NumberFormat(i18n.language, {
  style: 'unit', unit: 'watt', maximumFractionDigits: 1
}).format(watts)
```

### Adding a new locale

1. Create `src/locales/<code>/` and copy all JSON files from `en/` as a template
2. Translate all values (keys stay the same)
3. Add the code to `supportedLngs` in `src/services/i18n.ts`
4. Add an entry to `SUPPORTED_LOCALES` in `src/components/settings/LanguageSelect.tsx`
5. Add the native name to `settings.json` under `language`
6. Run `bunx i18next-parser` to verify completeness

### What NOT to do

```tsx
// ❌ Never — raw string in JSX
<Text>IP Address: {device.ip}</Text>

// ❌ Never — i18next-http-backend (requires HTTP server, breaks in Tauri)
import Backend from 'i18next-http-backend'
i18next.use(Backend)

// ✅ Correct
const { t } = useTranslation('devices')
<Text>{t('info.ipAddress')}: {device.ip}</Text>
```

## Common Pitfalls & Solutions

| Problem | Solution |
|---|---|
| HTTP request fails on mobile | Use `@tauri-apps/plugin-http`, not `window.fetch` |
| Routes show blank on deep link | Ensure `createHashHistory()` is used in router setup |
| Auto-generated route tree conflicts | Never manually edit `routeTree.gen.ts` |
| iOS can't reach LAN devices | Add `NSLocalNetworkUsageDescription` and `NSBonjourServices` to `Info.plist` |
| Android HTTP blocked | Add `network_security_config.xml` to allow cleartext traffic on LAN ranges |
| WS auth fails | Authenticate once per WS session, not per message |
| NotifyStatus breaks state | Deep-merge deltas, don't replace the whole status object |
| mDNS not finding devices | Browse `_shelly._tcp.local.` first (Gen2+ exclusive); also browse `_http._tcp.local.` as fallback |
| `routeTree.gen.ts` not updating | Ensure `TanStackRouterVite()` is the FIRST plugin in `vite.config.ts` before `react()` |
| Store not hydrated on first render | Call `deviceStore.hydrate()` in `__root.tsx` `useEffect` with empty dependency array |
| App blank screen on startup | Missing `<Suspense>` wrapper — the first locale chunk loads async; add `<Suspense fallback={<AppLoader />}>` around `<RouterProvider>` |
| Translation key returns key string | Namespace not loaded yet or key missing in JSON; check fallback log in dev console |
| TypeScript accepts wrong i18n key | `src/types/i18n.d.ts` not imported or locale JSON changed without restarting TS server |
| Language switch doesn't persist | `appStore.setLocale` must call both `i18next.changeLanguage()` AND `get().persist()` |
